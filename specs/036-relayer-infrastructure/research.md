# Phase 0 Research: Intent Relayer Infrastructure (Spec 036)

This document resolves the open unknowns for the Intent Relayer Infrastructure plan and feeds the plan's Technical Context and Structure Decision. Spec 036 makes the maintainer decision left open by spec 035: FairWins will operate a **self-hosted, open-source gas relayer** as a deliberate, documented, scoped exception to the fixed-footprint / no-backend directive, backstopped by the guaranteed self-submit path. The relayer must submit **both** spec-035 intent classes (EIP-3009 payment-carrying and no-stake signer-attributed) as bare signed calls from a managed EOA — not ERC-2771 forwarder meta-tx, not ERC-4337 account abstraction. Four research tracks were run to close the remaining unknowns; each resolved unknown is recorded below as Decision / Rationale / Alternatives considered, followed by a Technical Context resolution with concrete values and no remaining NEEDS CLARIFICATION.

Reference material: `specs/036-relayer-infrastructure/spec.md` (FR-001…FR-029, SC-001…SC-018), `specs/035-intent-based-payments/spec.md` (intent classes), and the existing reference relayer at `services/relayer/`.

---

## 1. OSS relayer selection and the gateway-vs-relayer architecture split

**Decision.** Adopt **OpenZeppelin Relayer** (the open-source Rust `openzeppelin-relayer`, AGPL-3.0) as the submission engine, fronted by a thin, FairWins-specific **relay gateway** that owns all intent- and signer-aware policy. Keep the existing `services/relayer/` Node/Express prototype only as the fallback baseline (its signer-screen + intent-binding logic migrates verbatim into the gateway); do not promote it to production. Designate **rrelayer** (Rust, MIT — https://github.com/joshstevens19/rrelayer) as the secondary engine if AGPL-3.0 becomes a licensing blocker.

The gateway ↔ engine boundary is drawn so the **gateway** owns everything that requires understanding a FairWins *intent* and *signer*, and **OZ Relayer** owns everything about getting a *transaction* mined; the gateway calls OZ Relayer's REST API (`POST /api/v1/relayers/{id}/transactions` with arbitrary `to`/`value`/`data`) and consumes its status webhooks. Both share one Redis.

- **Gateway (build — FairWins policy):** recover the EIP-712 / EIP-3009 signer; verify the intent binds target contract + params + validity window; **sanctions re-screen on the recovered signer, fail-closed** (FR-013, SC-005 — the engine's policy engine only sees the API caller, never the signer buried in calldata); **intent dedup on the uniqueness marker before submission** (FR-008, SC-006 — otherwise the engine happily re-broadcasts duplicate calldata under a fresh nonce and wastes gas); per-signer + global quotas and per-window spend cap in shared Redis (FR-014, FR-018, SC-012); fee-netting decline before any funds move (FR-023); back-pressure / bounded queue / near-expiry prioritization (FR-009); append-only audit intent→signer→txHash (FR-021); version-pinned addresses/ABIs + startup consistency check and kill switch (FR-015, FR-025); origin-lock validation behind the shared edge (FR-029).
- **OZ Relayer (configure, don't rebuild — submission mechanics):** per-`(chain, gas-wallet)` nonce lanes (FR-005); gas estimation + bump/replace of stuck tx with `gas_price_cap` (FR-006, SC-013); weighted `rpc_urls` failover (FR-007, SC-003); 5-stage lifecycle tracking to inclusion + webhooks (FR-006, FR-020); hot key in KMS/Vault + rotation (FR-017, FR-019, FR-019a); per-network config incl. **ETC legacy gas** (FR-022); persisted pending-nonce state + multi-instance Redis locks (FR-012, SC-008). Optionally pin OZ's per-relayer contract whitelist to the version-pinned target set as defense-in-depth.

**Rationale.** OZ Relayer is the only mature, self-hostable OSS project that submits **bare signed calls to an arbitrary target contract from a managed EOA** without imposing forwarder or smart-account semantics — exactly the primitive spec 035 requires, because both intent classes carry their authorization inside the calldata (EIP-3009 verified by the token; signer-attribution verified by the contract's EIP-712 check). It natively covers the mechanics the prototype lacks (nonce management, gas bumping, inclusion tracking, RPC failover, KMS-held signer, multi-chain JSON config), defaults to Redis (already budgeted by the spec's "operated compute plus a datastore," FR-004), and supports both EIP-1559 and legacy gas gated per-network by a `features: ["eip1559"]` flag — omitting that flag selects legacy pricing for ETC/Mordor, the capability managed relayers do not offer. Longevity is favorable: OpenZeppelin is sunsetting the managed **Defender** platform (final shutdown 2026-07-01) and explicitly steering users to self-host `openzeppelin-relayer` + `openzeppelin-monitor`, so the OSS relayer is the actively-invested path. The untrusted-by-design contract is preserved: the engine only submits what the gateway hands it and can censor but never steal (FR-003). Keeping the security-critical, auditable policy (~200 lines) in the FairWins-owned gateway in the repo's Node/ethers toolchain — while offloading the hard distributed-systems work to a maintained engine — is precisely why the prototype should not be scaled up in place.

**Alternatives considered.**
- **rrelayer (Rust, MIT):** feature-aligned (arbitrary EVM tx, configurable gas-bump cadence, per-user rate limiting, webhooks, low-balance alerts, AWS-KMS/Turnkey/Fireblocks/Privy/PKCS#11 signers). Kept as the escape hatch if AGPL-3.0 is unacceptable; rejected as primary only for being younger and less battle-tested.
- **Harden `services/relayer/` (Node/Express) as the engine:** rejected — would require re-implementing nonce lanes, gas bumping, inclusion tracking, RPC failover, KMS signing, persisted pending-nonce state, and multi-instance coordination, i.e. rebuilding OZ Relayer in Node. Its valuable part (signer sanctions re-screen + intent-binding checks in `services/relayer/src/relay.js`) is reused in the gateway instead.
- **Gelato Relay / Biconomy / OpenZeppelin Defender (managed SaaS):** ruled out — managed only (violates FR-001), ERC-2771/ERC-4337-oriented, no ETC/Mordor coverage; Defender is additionally being sunset 2026-07-01.
- **OpenGSN (OSS):** ruled out — architecture mismatch; requires targets to inherit `ERC2771Recipient` / use `_msgSender()`, and spec 035 explicitly rejects a shared meta-tx forwarder because nearly every existing check authorizes by direct caller.
- **ERC-4337 bundlers (Pimlico/Alto, Rundler, Skandha, Voltaire — OSS):** ruled out — process `UserOperation`s from smart-contract accounts via the EntryPoint; spec 035 puts AA/smart-wallet flows out of v1 scope (self-custody EOAs signing typed data). A bundler cannot submit a bare EIP-3009 call for an EOA.

## 2. ETC / Mordor gas, RPC, and USC / EIP-3009 gating

**Decision.** Configure ETC mainnet (chainId 61) and Mordor (chainId 63) as **legacy (type-0) gas-price chains** in the engine — omit the `eip1559` feature flag so all pricing and stuck-tx bumping use a single `gasPrice`, never `maxFeePerGas`/`maxPriorityFeePerGas`. Provision **≥2 independent RPC endpoints per chain** with failover (FR-007), and apply the repo's existing `batchMaxCount: 1` no-batch workaround to all 61/63 providers. Treat **gasless USC payment on ETC/Mordor as currently blocked**: the live USC token does **not** implement EIP-3009, so on 61/63 the relayer serves only no-stake signer-attributed intents (or self-submit) until an EIP-3009-capable USC exists; end-to-end legacy-gas submission on Mordor MUST be validated before it is relied upon.

**Rationale.**
- **Legacy gas only, pre-Cancun.** ETC deliberately never adopted EIP-1559 — Mystique (2022) shipped partial London EIPs but explicitly excluded the 1559 fee market; a base-fee treasury is only a proposal (Olympia) with no activation timeline. ETC is also pre-Cancun (Spiral hard fork, ~Feb 2024, brought only partial Shanghai — PUSH0, warm COINBASE, initcode metering; no `MCOPY`, no transient storage, no blob tx). Only type-0 transactions are available. The repo already encodes this: `hardhat.config.js:295-313` labels Mordor/etc as "Legacy chain (no EIP-1559)" with a `GAS_PRICE_WEI` pin, and notes the gas-price oracle suggests ~300 gwei. A relayer defaulting to type-2 fee logic will mis-price or fail on ETC.
- **RPC failover.** Mainnet (61) easily clears ≥2 independent providers: `https://etc.rivet.link` (repo default), `https://rpc.etcplanets.com`, the multi-client `etc-network.info` families, `https://etc.etcdesktop.com` (HebeBlock), NOWNodes. **Mordor (63) is thin — only two public endpoints exist** (`https://rpc.mordor.etccooperative.org` (repo default) + `https://geth-mordor.etc-network.info`), meeting the FR-007 minimum with zero slack; self-hosting a core-geth Mordor node as the second lane is advisable. **No-batch gotcha:** `frontend/src/utils/rpcProvider.js` sets `NO_BATCH_CHAIN_IDS = {61,63}` because the Caddy-fronted core-geth/besu endpoints return batch responses ethers v6 cannot decode; `services/relayer/src/chains.js` currently builds a plain `JsonRpcProvider` without `batchMaxCount: 1` and will hit this bug once 61/63 are enabled — the engine/gateway must mirror the workaround.
- **USC / EIP-3009 is absent on-chain (critical).** The repo records USC at `0xDE093684c796204224BC081f937aa059D903c52a` (decimals 6) for both 61 and 63 (`frontend/src/config/networks.js:118-123`, `deployments/mordor-chain63-v2.json`). On-chain verification (ETC mainnet Blockscout) confirms USC "Classic USD" is a UUPS proxy → `BraleToken`; its ABI supports **EIP-2612 `permit`** (`permit`/`nonces`/`DOMAIN_SEPARATOR`/`eip712Domain`) plus an `allow`/`deny`/`isAllowed` compliance blocklist, but has **no** `receiveWithAuthorization` / `transferWithAuthorization` / `cancelAuthorization` / `authorizationState`. EIP-3009 is therefore not merely unverified but absent — consistent with the gating dependency already flagged in `specs/035-intent-based-payments/spec.md`. The `services/relayer/` design (its `joinWithAuthorization` path) cannot work against real USC as deployed. Mordor (63) should still be read independently, though a Brale deterministic deploy almost certainly mirrors mainnet (permit-only). USC additionally carries a token-level allow/deny blocklist and is Brale-upgradeable, so a transfer can revert at the token even after the relayer's own SanctionsGuard passes — treat USC as a governed, not immutable, asset.
- **Operational.** ETC gas is paid in native ETC at a high legacy price (~300 gwei; a ~4.76M-gas deploy can exceed a node's 1-ETH per-tx fee cap per `hardhat.config.js:302-304`); thinner exchange liquidity than MATIC makes hot-key provisioning/refill more awkward — budget runway and a manual refill path (FR-018, FR-023). ETC uses chainId 61/63 for EIP-155 replay protection while core-geth reports networkId 1 on mainnet; keep signing keyed to the true chainId (`chains.js` already pins `staticNetwork: Network.from(chainId)`).

**Alternatives considered.**
- **EIP-2612 `permit` + pull fallback for USC on 61/63:** technically available on the live token, but spec 034 research §5 explicitly de-preferred permit for its different signing domain, sequential nonces, and front-runnable/griefable approval — recorded as a possible fallback, not adopted.
- **Confirm/commission an EIP-3009 (FiatToken-style) USC from Brale:** the clean unblock for gasless USC payment, but out of FairWins' control; tracked as the gating dependency rather than assumed.
- **Managed relayer for ETC:** ruled out upstream — managed services generally do not serve ETC/Mordor (a primary reason for self-hosting, FR-022).

Sources: https://ethereumclassic.org/blog/2024-07-24-the-negative-effects-of-eip-1559/ · https://ecips.ethereumclassic.org/ECIPs/ecip-1109 · https://ethereumclassic.org/network/endpoints/ · https://brale.xyz/stablecoins/USC · ETC Blockscout on-chain read (USC proxy → `BraleToken`, permit-yes / EIP-3009-no) · repo: `frontend/src/config/networks.js`, `frontend/src/config/contracts.js`, `hardhat.config.js:295-313`, `frontend/src/utils/rpcProvider.js`, `deployments/mordor-chain63-v2.json`, `services/relayer/src/chains.js`.

## 3. Persistence strategy and the honest net-new-infrastructure list (FR-004 decision record)

**Decision (per state class, footprint-minimizing).**
- **Audit trail (FR-021 / SC-017, ≥5yr, append-only):** reuse **Cloud Logging** as the write path (structured JSON audit events — already the mandated sink per `cloudbuild.yaml` `CLOUD_LOGGING_ONLY`), routed via a **Log Router sink to a retention-locked destination**. Preferred: a **GCS bucket with Bucket Lock (WORM)** + a lifecycle rule to Coldline/Archive; zero-new-resource fallback: a dedicated **Cloud Logging log bucket with a locked retention policy** (`gcloud logging buckets update <bucket> --retention-days=1825 --locked`; up to 3650 days supported). **No new database.**
- **Operational shared state (nonce lanes, dedup, rate-limit/spend counters — FR-012 / FR-014 / SC-012):** phased. **Phase 1 (v1):** single Cloud Run instance (`--max-instances=1`), state in-process memory — FR-012's shared-store requirement is explicitly conditional ("When run as multiple instances"), and SC-001 volumes (≥3 intents/sec/chain) fit one instance, adding zero net-new datastore; restart loss is bounded (dedup is protocol-safe via the on-chain uniqueness marker, rate-limit reset is a benign momentary loosening). **Phase 2 (only when horizontal scale is needed):** add **Memorystore for Redis, Basic tier** (single node, no HA) — atomic `INCR`+TTL for cross-instance counters/spend caps and nonce coordination; Basic tier matches the best-effort/no-SLA posture (self-submit is the availability backstop, so HA Redis would over-invest).
- **Pending-nonce across deploys (FR-025):** **reconstruct from chain on startup** — per `(chain, gas-wallet)` lane, read `eth_getTransactionCount(wallet, "pending")` and resume, combined with the FR-005 serialized single-writer lane + gap/stuck recovery. Redis (Phase 2) holds only a rebuildable in-flight cache, never the system of record — deliberately avoiding Redis AOF/persistence and a higher Memorystore tier. Graceful drain (FR-025) minimizes the in-flight set at deploy time.
- **Hot key at rest (FR-019a):** **GCP Secret Manager**, injected at Cloud Run runtime identically to `PINATA_JWT` / `ORIGIN_LOCK_SECRET`, least-privilege bound to the relayer service account; never the `.env` plaintext pattern; never logged/telemetered/audited. Optional hardening: a Cloud KMS-held key with remote signing (key never leaves KMS).

**Rationale.** The guiding principle is to honor the fixed-footprint / no-backend directive (`docs/developer-guide/architecture.md:163-165`) as far as the relayer exception allows: on-chain (`txHash`) is the permanent record of record, and every off-chain store is a rebuildable convenience layer, never the source of authority. The audit record is derivable metadata whose authoritative signer→txHash binding is already permanent on-chain, so Cloud Logging + WORM retention meets the AML-grade append-only requirement with no database. Reconstruct-from-chain removes any need for Redis durability, keeping Memorystore at Basic tier and deferrable. Secret Manager is the exact fit FR-019a names and reuses a proven injection path with no new resource type. The spec itself scopes the exception to "operated compute plus a datastore" (FR-003/FR-004, Key Entities), so Memorystore Basic is in-budget precisely and only as that datastore.

**Net-new infrastructure (honest scope).**

| # | Item | Net-new? | Notes |
|---|------|----------|-------|
| 1 | Relayer/gateway service on **Cloud Run** | **Net-new operated compute** | The core of the FR-004 exception; reuses the Cloud Run platform but is a new always-reachable service. |
| 2 | **Memorystore for Redis, Basic** | **Net-new datastore — Phase 2 only** | The single datastore the exception's "plus a datastore" refers to; not needed for single-instance v1. |
| 3 | **Serverless VPC Access connector** | **Net-new networking — Phase 2 only** | Ships only with item 2 (Cloud Run → Memorystore private IP). |
| 4 | GCS bucket w/ Bucket Lock (audit archive) | Storage-only, optional | Zero compute; avoidable by using a locked Cloud Logging log bucket instead. |
| 5 | New Secret Manager secret (gas hot key) | Reuses Secret Manager | One new secret entry, no new infra type. |
| 6 | Cloudflare route/hostname for the relayer | Reuses Cloudflare | Config only (see §4). |

**Honest bottom line:** the minimum viable footprint adds exactly **one net-new thing — the operated relayer compute on Cloud Run.** Scaling past one instance adds **one managed datastore (Memorystore Basic) + one VPC connector.** Everything else (audit, pending-nonce, hot key, edge perimeter) is absorbed by existing components.

**Alternatives considered.**
- **Audit in a new managed DB (Cloud SQL / Firestore / BigQuery):** rejected — a net-new database for append-only, write-once, rarely-read compliance metadata already anchored on-chain; BigQuery remains available as an additive downstream analytics sink off the same Log Router if querying is ever needed.
- **Memorystore Standard/HA with persistence as primary shared+nonce store:** rejected — contradicts fixed-footprint and best-effort posture; reconstruct-from-chain removes the need for Redis durability.
- **Redis as a Cloud Run sidecar:** rejected — Cloud Run instances are ephemeral and independently scaled, so a sidecar is not shared across instances and fails FR-012.
- **KMS remote-signing instead of Secret Manager:** viable stronger variant, deferred as optional hardening since the key is deliberately low-value/capped and Secret Manager reuses a proven path.

Cross-links for the FR-004 record: overrides `docs/developer-guide/architecture.md:163-165` and the spec 007 "no backend, no new compute" clarification; **supersedes spec 035 SC-008 and resolves spec 035 FR-017**.

## 4. Deployment and Cloudflare perimeter topology (FR-029)

**Decision.** Run the relayer/gateway as a **separate Cloud Run service on a proxied subdomain in the existing `fairwins.app` Cloudflare zone** (`relayer.fairwins.app`, orange-cloud), and enforce origin-lock **inside the relayer process** via a small `X-Origin-Auth` timing-safe-compare middleware in the gateway's `server.js`, reusing the **same** `origin-lock-secret` and the **same** zone-wide WAF geo rule + Transform Rule that already protect the SPA. Do not stand up a second edge, a new GCP load balancer, or Cloud Armor. Recommended fallback (lowest new-config): expose the relayer as a **path under the SPA host** (`https://fairwins.app/relay/*`) via a new nginx `location /relay/` that reuses the existing `$origin_denied` guard and `proxy_pass`es to the relayer — no new hostname, no CSP change.

**Rationale.** The SPA perimeter has three layers the relayer must inherit, not reinvent: (1) a **WAF geo rule** on `ip.src.country` returning **HTTP 451** with `frontend/public/451.html` (`infra/cloudflare/waf-geo.md`); (2) an **origin-lock** Transform Rule that injects `X-Origin-Auth: <secret>` on **all incoming requests** zone-wide, with the secret in Secret Manager as `origin-lock-secret` (`infra/cloudflare/origin-lock.md`); (3) **origin verification** where `frontend/nginx.conf.template` maps the header to `$origin_denied` → 403 in every location (`/healthz` exempt). Geo is enforced at the edge on the true client IP before the origin, and the Transform Rule already matches "All incoming requests," so a proxied same-zone relayer host is 451'd for restricted regions automatically and needs no relayer-specific geo logic. The relayer today (`services/relayer/src/server.js`) is a bare Express app with **no `X-Origin-Auth` check** and would get a public `*.run.app` URL — exactly the direct-origin bypass origin-lock exists to close (SC-016). Enforcing the ~10-line header check in the relayer itself (mirroring the nginx `map`, exempting `/healthz`, active only when the secret is set) avoids an extra nginx hop since Cloud Run already terminates TLS, and keeps the security-critical check in the same auditable file. A raw `*.run.app` hit without the header returns 403; the bypass is closed.

Deployment steps: (1) deploy the relayer as its own Cloud Run service (`fairwins-relay-gateway`, `--allow-unauthenticated`, us-central1) from its Dockerfile, a sibling of `cloudbuild.yaml`'s existing deploy step; (2) add proxied DNS `relayer.fairwins.app`; (3) confirm the WAF geo rule is host-agnostic or widen its `http.host` scope to include the new host, keeping the 451 response; (4) the Transform Rule needs no change; (5) add the `X-Origin-Auth` middleware and wire `--update-secrets ORIGIN_LOCK_SECRET=origin-lock-secret:latest`; (6) verify per `origin-lock.md` (direct `*.run.app` → 403; wrong header → 403; via subdomain → 200; `/healthz` → 200; restricted region → 451). Client wiring: `frontend/src/lib/pools/relayerClient.js` reads `VITE_POOL_RELAYER_URL` (inlined at build time, **currently absent** from `Dockerfile` / `cloudbuild.yaml` / `frontend/.env.example`); subdomain topology sets it to `https://relayer.fairwins.app` and **must add that origin to the SPA CSP `connect-src` in `nginx.conf.template`** (cross-origin — not there today, so the browser blocks the call until added). Path topology sets it to `https://fairwins.app` (same-origin, allowed by `connect-src 'self'`, no CSP edit). Either way the "unset → `makePoolRelayer()` returns null → normal gas-paying self-submit" fallback is preserved (`relayerClient.js`), so the perimeter change never makes gasless a hard dependency.

**Alternatives considered.**
- **Same host, path prefix (`fairwins.app/relay/*`) via nginx proxy:** reuses the audited `$origin_denied` guard, zero new Cloudflare objects, same-origin (satisfies existing CSP) — modeled on the existing `location /api/pinata/` block. Rejected as *primary* only because it couples relayer traffic to the SPA container and adds a proxy hop; kept as the recommended fallback.
- **New GCP ALB + Cloud Armor + mTLS (Authenticated Origin Pulls):** strongest cryptographic origin lock, but out of scope — adds new infra and Cloud Armor cannot emit 451; documented as future hardening.
- **Bare Cloudflare-egress-IP allowlist:** rejected for the same reason the SPA rejects it — those IPs are shared across all Cloudflare tenants and can spoof geo headers.
- **Third-party managed relayer:** moves the endpoint outside the FairWins zone, cannot inherit this perimeter, and would be a geo/compliance bypass — incompatible with FR-029.

## Technical Context resolution

All NEEDS CLARIFICATION items are resolved to concrete values below.

**Language/Version:** Two-component service. (a) **Relay gateway** — Node.js 20 LTS + ethers v6 (matches the repo toolchain and the existing `services/relayer/` prototype; Express + helmet). (b) **Submission engine** — OpenZeppelin Relayer, Rust (pinned 1.x release), run as a configured container, not a code dependency. Contracts remain Solidity (unchanged by this feature; the signer-attributed entrypoints they call are delivered by spec 035).

**Primary Dependencies:** OpenZeppelin Relayer (`openzeppelin-relayer`, AGPL-3.0, version-pinned) as the submission engine + `openzeppelin-monitor` for FR-020 observability; Node/Express + ethers v6 for the gateway; Redis (via Memorystore, Phase 2) as the shared nonce/queue/lock + dedup/quota/spend-cap store; GCP Secret Manager (hot key + origin-lock secret); GCP Cloud Logging + Log Router; Cloudflare (WAF geo rule + Transform Rule origin-lock). Reuses the on-chain `ISanctionsGuard` (signer screening) and version-pinned target-contract ABIs. Secondary/fallback engine: rrelayer (MIT). Not used: ERC-2771 forwarders, ERC-4337 bundlers/EntryPoint.

**Storage:** On-chain is the permanent record of record. Off-chain: **audit trail** → Cloud Logging → Log Router sink → WORM destination (GCS Bucket Lock preferred, or a locked Cloud Logging log bucket), retained ≥5 years. **Operational shared state** (nonce lanes, dedup, rate-limit/spend counters) → in-process memory in Phase 1 (single instance), **Redis (Memorystore Basic)** in Phase 2. **Pending-nonce** → reconstructed from chain on startup (`eth_getTransactionCount(..,"pending")`); Redis holds only a rebuildable cache. **Hot key** → GCP Secret Manager (optional Cloud KMS remote-signing). No relational/document database.

**Testing:** Hardhat/Vitest for the repo's existing suites; **gateway** unit + integration tests (Vitest/Node, ethers) for signer recovery, sanctions fail-closed, dedup idempotency, quota/spend-cap, fee-netting decline; **contract/integration** tests against a staging chain for nonce-collision-free submission, stuck-tx re-price, RPC failover, self-submit fallback, kill switch, multi-instance (≥2) nonce/limit correctness; **load test** — a repeatable, **CI-gating** harness (FR-011/SC-001/SC-002) using **k6** (scriptable sustained + burst profiles, thresholds that fail the pipeline on latency/throughput regression) driving the gateway against a test/staging chain; ETC/Mordor legacy-gas submission validated end-to-end before reliance.

**Target Platform:** Linux containers on **GCP Cloud Run** (us-central1), behind the existing Cloudflare edge (`fairwins.app` zone). Submits to Polygon (137), Amoy (80002), and Ethereum Classic mainnet (61) / Mordor (63).

**Project Type:** Self-hosted backend microservice (operated compute) — a documented, scoped exception to the no-backend directive — composed of a FairWins-owned relay gateway fronting an OSS submission engine; additive to the existing SPA + contracts + subgraph, never a single point of failure (self-submit backstop).

**Performance Goals:** Sustain **≥3 intents/sec/chain**; burst **≥20 intents/sec for 30s** across configured chains (SC-001, tunable knobs, not ceilings); **p95 accept→broadcast latency <2s** at target load (SC-002); **zero transaction-nonce collisions** including with ≥2 instances (SC-001/SC-012); RPC-failover error rate back to baseline within **30s** and **<1%** during failover (SC-003); stuck tx re-priced and included within a bounded window without wedging the lane (SC-013); designed to scale horizontally to raise these targets.

**Constraints:** Untrusted-by-design — can censor, never steal, alter, mis-attribute, or trap funds (FR-003); holds no user funds and no contract authority; screens the **signer** and fails closed (FR-013); optional/additive with **100% self-submit fallback** within bounded time-to-fallback (FR-002/SC-004); hot key is dedicated, low-value, spend-capped (per-window rate cap + absolute per-key exposure cap), KMS/Secret-Manager-held, segregated from air-gapped floppy admin keys, never logged, rotatable with zero lost/double-submitted intents; ETC/Mordor is **legacy (type-0) gas only, pre-Cancun**, with the `batchMaxCount:1` RPC workaround, and gasless USC payment on 61/63 is **blocked** pending an EIP-3009-capable token (no-stake signer-attributed intents and self-submit remain available); addresses/ABIs version-pinned with a startup consistency check; graceful drain on deploy; network isolation (no testnet/mainnet crossover); behind the same Cloudflare 451 geo-gate + origin-lock as the SPA (FR-029); AGPL-3.0 network-copyleft managed by keeping FairWins changes in the separately-licensed gateway (rrelayer/MIT is the fallback); best-effort availability, no hard uptime SLA; CI gating with no `continue-on-error`; WCAG 2.1 AA for the new user-facing states.

**Scale/Scope:** Early-stage with headroom by design. Four networks (Polygon 137, Amoy 80002, ETC 61, Mordor 63), one independent nonce lane + gas wallet per chain. Both spec-035 intent classes in v1 (EIP-3009 payment-carrying + no-stake signer-attributed: claim, refund, cancel/decline, draw propose/approve/revoke, voucher redeem, pool propose/vote/claim) via a generic signer-attributed submission path. Phase 1 single Cloud Run instance; Phase 2 horizontal scale with shared Redis when volumes exceed single-instance capacity. Audit retention ≥5 years. Net-new footprint: exactly one operated compute service (Cloud Run) in v1; +Memorystore Basic +VPC connector when scaling past one instance.
